# Issue #001 背景ノードの扱い

## 現状

背景は255件まで登録可能としているが、
画面上でどのように配置・表示するかが未定義。

## 決定

案1を採用する。背景ノードを最上位としてキャンバスに表示し、
各上位要求は必ず1つの背景ノードに属するものとする。

上位要求に複数の背景ノードを親として設定することは認めない。

## 候補

### 案1

背景ノードを最上位として表示する。

```
背景
 ↓
上位要求
```

### 案2

背景ノードを独立して複数表示する。

```
背景1
背景2
背景3

 ↓

上位要求
```

### 案3

背景は画面左上に一覧表示し、
グラフには表示しない。

## 優先度

高

## 状態

決定済み

# Issue #003 仕様の共有

## 現状

仕様は1つの下位要求に属する前提。

しかし

```
要求A
   \
    \
    仕様
    /
要求B
```

というケースが存在する。

## 決定

仕様は必ず1つの下位要求に属するものとする。
複数の下位要求を親として設定することは認めない。

所属先以外の下位要求との関係は、所有関係ではなく任意の関連情報として保存できる。
YAML では `related_requirement_ids` に関連する下位要求 ID を配列で保存する。
この関連情報は仕様の親子関係や ID 採番を変更しない。

```yaml
specifications:
  - id: "001-001-001"
    specification: 編集内容を一定時間ごとに保存する
    parent_requirement_id: "001-001"
    related_requirement_ids:
      - "002-001"
```

## 検討事項

仕様を共有可能にするか。

## 優先度

高

## 状態

決定済み

# Issue #013 ノード削除時の子要素の扱い

## 現状

ノード削除時に、配下の下位要求や仕様をどう扱うか未定。

## 決定

親ノードを削除する場合は、配下の子要素をすべて削除する。
削除前には確認ダイアログを表示し、削除対象の件数を示す。

確認ダイアログには「今後この確認を表示しない」チェックボックスを設ける。
チェックした場合、同じブラウザでは次回以降の削除確認を表示しない。

## 検討事項

- 親ノード削除時に子ノードも削除するか
- 子ノードを別の親へ移動できるようにするか
- 削除前に確認ダイアログを表示するか

## 推奨

初期実装では、親ノード削除時に配下要素も削除する。

削除前に確認ダイアログを表示し、削除対象件数を表示する。

## 優先度

高

## 状態

決定済み

# Issue #014 理由を必須項目にするか

## 現状

要求仕様書では要求ノードの項目として「理由」が定義されているが、入力必須かどうかは未定。

## 決定

理由は必須項目としない。
YAML 保存時は理由が空でも警告を表示せずに保存する。

USDM 表、Excel などのファイルエクスポート時は、理由が空の要求を警告として表示する。
この警告はエクスポートを妨げず、ユーザーは内容を確認したうえで出力を続行できる。

## 検討事項

- 理由を必須にするか
- 空欄を許可するか
- 空欄の場合に警告だけ出すか

## 推奨

USDM の性質上、理由は必須とする。

ただし、編集中の一時保存では空欄を許可し、保存前検証で警告またはエラーにする案も検討する。

## 優先度

中

## 状態

決定済み

# Issue #021 Git差分を考慮した保存

## 現状

Git管理を前提としている。

## 決定

変更差分を最小化するため、YAML は種別ごとのフラットな一覧構造で保存する。
シリアライザーは、トップレベルキー、各ノードの項目、配列を固定順で出力する。

- トップレベルキー: `version`、`title`、`backgrounds`、`requirements`、`specifications`
- ノード一覧: ID 昇順
- ノード項目: ID、本文、親 ID、関連 ID の順
- `related_requirement_ids`: ID 昇順
- 任意項目が空の場合: 出力しない
- YAML のインデント: 半角スペース2個、改行コード: LF
- 保存日時や自動生成 ID など、内容と無関係な情報は保存しない

## 検討事項

変更差分が最小になるYAML構造を設計する。

## 優先度

高
